Campaign Builder Wizard — Architecture Approaches

How should the wizard interact with the backend during creation?

A

Eager Save (Recommended)

Create the campaign immediately on "Create New Campaign" click. Each wizard tab saves data to the backend as you go — locations, phases, NPCs are created via existing API endpoints as you fill them in.

Pros

  • Reuses all existing CRUD endpoints — no new backend routes needed for wizard
  • No data loss if browser closes mid-wizard
  • Locations/NPCs/PCs created during wizard are immediately available
  • Simple mental model: wizard is just a guided UI over existing APIs
  • Overview page editing works automatically — same data, same endpoints

Cons

  • Abandoned wizards leave empty/partial campaigns (needs cleanup or draft state)
  • Tab navigation requires saving before switching (or auto-save)
B

Deferred Save (All-at-Once)

Collect all wizard data in frontend state. On final "Finish" step, send everything to a new dedicated backend endpoint that creates campaign + all related entities in one transaction.

Pros

  • No partial campaigns — either everything saves or nothing does
  • Clean cancel behavior — just navigate away, nothing to clean up

Cons

  • Requires a complex new "bulk create" backend endpoint
  • All data lost if browser closes mid-wizard
  • Large frontend state management for all entities across tabs
  • Location tree with temp IDs needs mapping on save
  • Can't reference real entity IDs until final save
C

Hybrid — Draft Campaign + Eager Entities

Create campaign immediately with a status: "draft" field. Entities save eagerly as in Option A. Draft campaigns are hidden from the main list until the wizard is completed, at which point status flips to "active".

Pros

  • Clean campaign list — no half-built campaigns showing up
  • All the eager-save benefits (no data loss, reuse existing APIs)
  • Clear "wizard complete" transition point

Cons

  • New status field on campaign model + migration
  • Filter logic needed on campaign list queries
  • Slightly more backend work than pure Option A
